# Как настроить Sentry вместе с Leaf

[Sentry](https://sentry.io/) — это мощный сервис для отслеживания ошибок в реальном времени и мониторинга производительности. Его интеграция с вашим сервером Leaf позволяет автоматически перехватывать ошибки и исключения, что значительно упрощает диагностику и устранение проблем на сервере.

Это руководство объясняет, как настроить Leaf для отправки отчётов об ошибках в ваш проект Sentry.

## Предварительные требования

1.  **Аккаунт Sentry:** Создайте аккаунт на [sentry.io](https://sentry.io/).
2.  **Проект Sentry:** Создайте новый проект в вашей организации Sentry.
    - При выборе платформы выберите **Java**. Если Java не отображается сразу, выберите "Other" или воспользуйтесь поиском. Выбор конкретной платформы в основном влияет на инструкции по начальной настройке на стороне Sentry, но Leaf сам управляет интеграцией.
3.  **DSN (Data Source Name):** После создания проекта перейдите в его настройки. Найдите раздел "Client Keys (DSN)". Скопируйте строку DSN — она выглядит как URL (например, `https://xxxxxxxxxxxxxxxxxxxxxxxx@o######.ingest.sentry.io/#######`).

## Шаги настройки

1.  **Найдите файл конфигурации Leaf:** Откройте основной файл конфигурации вашего сервера Leaf. Обычно это `leaf-global.yml`, расположенный в корневой директории сервера или в подпапке `config`.

2.  **Найдите секцию Sentry:** Внутри файла конфигурации найдите секцию `sentry:`. По умолчанию она выглядит примерно так:

    ```yaml
    sentry:
        # Sentry DSN for improved error logging, leave blank to disable,
        # Obtain from https://sentry.io/
        dsn: ""
        # Logs with a level higher than or equal to this level will be recorded.
        log-level: WARN
        # Only log with a Throwable will be recorded after enabling this.
        only-log-thrown: true
    ```

3.  **Настройте параметры:**
    - **`dsn`:**
        - Замените пустые одинарные кавычки (`''`) на DSN, скопированный из настроек вашего проекта Sentry. Убедитесь, что он заключён в одинарные кавычки.
        - **Пример:** `dsn: 'https://xxxxxxxxxxxxxxxxxxxxxxxx@o######.ingest.sentry.io/#######'`
        - _Если оставить это поле пустым, интеграция с Sentry будет отключена._

    - **`log-level`:**
        - Определяет минимальный уровень серьёзности сообщений журнала, которые Leaf будет отправлять в Sentry. Распространённые уровни: `ERROR`, `WARN`, `INFO`, `DEBUG`.
        - Значение по умолчанию `WARN` означает, что будут отправляться только сообщения уровня предупреждения или ошибки (которые, как правило, включают исключения).
        - Можно установить `ERROR`, если нужны только критические ошибки. Установка более низкого уровня (например, `INFO`) приведёт к отправке значительно большего объёма данных, что нежелательно, если вы не занимаетесь активной отладкой.
        - **Рекомендация:** Начните с `WARN` (значение по умолчанию).

    - **`only-log-thrown`:**
        - Если установлено значение `true` (по умолчанию), Leaf будет отправлять в Sentry _только_ те сообщения журнала, которые явно содержат Java `Throwable` (например, исключение или ошибку со стек-трейсом). Это настоятельно рекомендуется, поскольку позволяет сосредоточить Sentry на реальных ошибках кода.
        - Если установлено значение `false`, в Sentry будет отправляться _любое_ сообщение журнала, соответствующее порогу `log-level`, даже если это просто информационный текст без ошибки. Это может создать много шума в Sentry.
        - **Рекомендация:** Оставьте значение `true` (по умолчанию).

4.  **Сохраните и перезапустите:**
    - Сохраните изменения в файле конфигурации.
    - Полностью перезапустите сервер Leaf, чтобы новые настройки вступили в силу.

## Устранение неполадок

- **Ошибки не отображаются:**
    - Убедитесь, что `dsn` скопирован правильно и заключён в одинарные кавычки в YAML-файле.
    - Проверьте, что `log-level` не установлен слишком высоко (например, если установлен `ERROR`, предупреждения отправляться не будут).
    - Убедитесь, что `only-log-thrown` соответствует типу ожидаемых ошибок.
    - Убедитесь, что сервер был полностью перезапущен после изменения конфигурации.
    - Проверьте, не блокирует ли брандмауэр вашего сервера исходящие соединения к `ingest.sentry.io` на порту 443 (HTTPS).
- **Слишком много проблем:**
    - Повысьте `log-level` (например, с `WARN` до `ERROR`).
    - Убедитесь, что `only-log-thrown` установлен в `true`.
